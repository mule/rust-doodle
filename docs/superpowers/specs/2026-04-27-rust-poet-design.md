# rust-poet — Design

**Date:** 2026-04-27
**Status:** Approved (brainstorming complete)
**Location:** `cross-platform/rust-poet/`

## 1. Purpose & scope

`rust-poet` is a sandbox project to learn how to:

1. Build a small Rust **service** (here: a library + thin CLI) that integrates third-party HTTP APIs.
2. Wire up **multiple LLM providers** behind a single abstraction, selectable at runtime via config.
3. Pull material for generation from another **third-party API** (Wikipedia "On this day").

Concrete goal of the proof of concept: run `rust-poet` from the terminal and get back a poem about a topic — given, random, or derived from today's historical events — generated by the LLM provider configured by the user.

### In scope (PoC)

- Library + binary crate at `cross-platform/rust-poet/`
- `LlmProvider` trait with three impls: OpenAI, Anthropic, Mistral
- `TopicSource` trait with three impls: fixed (`--topic`), random (built-in seed list), Wikipedia "On this day"
- Layered configuration: built-in defaults → RON file → CLI overrides; API keys via env only
- Structured error handling (`thiserror` for the library, `anyhow` for the binary)
- Unit + integration + (opt-in) live tests

### Explicitly out of scope (deferred to phase 2+)

- Bevy-based rendering with shader effects
- Text-to-speech / voice generation
- Streaming token output
- Retry / backoff on rate-limit responses
- Response caching
- Style / mood / form modifiers (`--style sonnet`, `--mood melancholic`)
- News / RSS / themed-list topic sources
- Additional providers (Gemini, Cohere, …)

A separate "deferrals" table at the end of this document records how the design accommodates each future item.

## 2. Architecture

Single crate, library-first. The binary (`src/main.rs`) is a thin CLI wrapper; **all real logic lives in the library** (`src/lib.rs` and modules) so a future Bevy crate can `use rust_poet::*` directly without a process boundary.

### 2.1 Module layout

```
cross-platform/rust-poet/
├── Cargo.toml
├── poet.example.ron          # committed sample config
├── src/
│   ├── lib.rs                # re-exports public API
│   ├── main.rs               # clap parsing, calls into lib, prints poem
│   ├── config.rs             # RON loader + CLI/env merging
│   ├── error.rs              # PoetError (thiserror)
│   ├── poet.rs               # Poet engine — orchestrates the pipeline
│   ├── prompt.rs             # builds Vec<Message> from a Topic
│   ├── provider/
│   │   ├── mod.rs            # LlmProvider trait + factory + ProviderError
│   │   ├── openai_like.rs    # shared request/response shapes
│   │   ├── openai.rs
│   │   ├── mistral.rs
│   │   └── anthropic.rs
│   └── topic/
│       ├── mod.rs            # TopicSource trait + factory + TopicError
│       ├── fixed.rs          # --topic "string"
│       ├── random.rs         # built-in seed list
│       └── wikipedia.rs      # "On this day"
└── tests/
    └── ...                   # integration tests using MockLlmProvider
```

### 2.2 Units & boundaries

| Unit | Purpose | Depends on |
|---|---|---|
| `LlmProvider` (trait) | Send messages → get text back | nothing |
| `TopicSource` (trait) | Produce a `Topic` to write about | nothing |
| `prompt::build` | `Topic` → `Vec<Message>` | `Topic` only |
| `Poet` | Compose source + prompt + provider into one `generate()` call | the three above |
| `config` | Load + merge config layers | nothing app-specific |
| `main.rs` | Parse CLI args, call `Poet::generate`, print | everything (only here) |

The two traits are the design's load-bearing seams. They keep providers and topic sources independently testable, swappable, and addable. `Poet` is provider- and source-agnostic; it just orchestrates.

## 3. Public types & trait shapes

```rust
// --- provider/mod.rs ---
#[async_trait::async_trait]
pub trait LlmProvider: Send + Sync {
    fn name(&self) -> &'static str;
    async fn generate(&self, req: &LlmRequest) -> Result<LlmResponse, ProviderError>;
}

pub struct LlmRequest {
    pub messages: Vec<Message>,
    pub model: String,
    pub max_tokens: u32,        // non-optional: Anthropic requires it
    pub temperature: f32,
}

pub struct Message { pub role: Role, pub content: String }
pub enum Role { System, User } // Assistant role unused in PoC

pub struct LlmResponse {
    pub text: String,
    pub model: String,          // echo of the model that actually answered
    pub usage: Option<Usage>,
}
pub struct Usage { pub prompt_tokens: u32, pub completion_tokens: u32 }

// --- topic/mod.rs ---
#[async_trait::async_trait]
pub trait TopicSource: Send + Sync {
    fn name(&self) -> &'static str;
    async fn next_topic(&self) -> Result<Topic, TopicError>;
}

pub struct Topic {
    pub seed: String,           // short tag, e.g. "rain" or "In 1889, ..."
    pub context: Option<String>,// optional background material for the prompt
}

// --- poet.rs ---
pub struct Poet { /* provider, source, settings */ }

impl Poet {
    pub fn new(
        provider: Box<dyn LlmProvider>,
        source: Box<dyn TopicSource>,
        settings: PoemSettings,
    ) -> Self;

    pub async fn generate(&self) -> Result<Poem, PoetError>;
}

pub struct PoemSettings { pub model: String, pub max_tokens: u32, pub temperature: f32 }

pub struct Poem {
    pub text: String,
    pub topic: Topic,
    pub provider: &'static str,
    pub model: String,
}
```

### 3.1 Notable design decisions

1. **Dynamic dispatch (`Box<dyn …>`) over generics.** Both the provider and the topic source are picked at runtime from config — generics would force the choice into the type of `Poet`, which is awkward when `main.rs` is doing `match config.provider { … }`. The cost (one v-table indirection per request) is irrelevant for an HTTP-bound op. Trade-off: requires `async_trait` macro, since native async-fn-in-trait isn't object-safe yet.

2. **`max_tokens: u32` (non-optional)** in `LlmRequest`. Anthropic's API requires it; making it non-optional in the trait surface means we can never call any provider without setting it, and we never need a special case at the call site.

3. **`Topic.context` is the seam for "events of the day."** Wikipedia returns rich material per day — pinning it as a separate field (rather than mashing it into `seed`) lets the prompt builder decide how much of it to inject.

4. **`LlmResponse.model` echoes back what answered.** Useful when an API silently substitutes (e.g., a "latest" alias points somewhere new), and essential when phase 2 wants to display "generated by claude-…" alongside the poem.

## 4. Provider implementations

### 4.1 Wire-format differences

|  | OpenAI | Mistral | Anthropic |
|---|---|---|---|
| Endpoint | `api.openai.com/v1/chat/completions` | `api.mistral.ai/v1/chat/completions` | `api.anthropic.com/v1/messages` |
| Auth header | `Authorization: Bearer …` | `Authorization: Bearer …` | `x-api-key: …` + `anthropic-version: 2023-06-01` |
| System prompt | inside `messages` (`role: "system"`) | inside `messages` | **top-level `system` field** |
| `max_tokens` | optional | optional | **required** |
| Response path | `choices[0].message.content` | same | `content[0].text` |
| Error envelope | `error.{message, type, code}` | same | `error.{type, message}` |

OpenAI and Mistral are protocol-twins; Anthropic is the genuine outlier.

### 4.2 Internal code-sharing structure

`provider/openai_like.rs` exposes shared request/response structs and a `send()` helper. `openai.rs` and `mistral.rs` are ~30-line files that delegate to it with different `base_url`, env-var name, and default model. `anthropic.rs` is its own ~80-line impl.

The shared helper is **internal** — the public abstraction is the `LlmProvider` trait. This separation matters: it would be a mistake to leak provider quirks (e.g., a `system_message` field as if Anthropic's top-level field were universal) into the trait.

Adding a fourth OpenAI-compatible provider (Groq, Together, a local Ollama in OpenAI-compat mode) is a 30-line file. Adding a genuinely-different provider (Gemini) is its own ~80-line file.

### 4.3 ProviderError

Every impl maps its provider-specific HTTP/JSON error into one of:

```rust
pub enum ProviderError {
    Network(reqwest::Error),                                // connection/timeout/DNS
    Auth { provider: &'static str },                        // 401/403
    RateLimited { retry_after: Option<Duration> },          // 429
    BadRequest { provider: &'static str, message: String }, // 400 — bad model name, etc.
    ServerError { provider: &'static str, status: u16 },    // 5xx
    EmptyResponse { provider: &'static str },               // 200 but no text
    Decode(serde_json::Error),
}
```

Mapping happens at the edge of each impl. `Poet` and the CLI never see provider-specific error types.

## 5. Topic sources

### 5.1 Fixed source

Wraps the `--topic <string>` argument. `next_topic()` returns `Topic { seed: arg.clone(), context: None }`. Trivial.

### 5.2 Random source

Built-in static seed list (10–20 evocative entries: "the sea", "machinery", "loneliness", "morning light", …). `rand::seq::SliceRandom::choose` picks one. No network. Ships in the binary.

### 5.3 Wikipedia "On this day" source

- **Endpoint:** `GET https://en.wikipedia.org/api/rest_v1/feed/onthisday/events/{MM}/{DD}` — no auth, no key.
- **Date:** `chrono::Local::now().date_naive()`.
- **Response shape:** `{ events: [{ text, year, pages: [{ extract, … }] }, …] }`.
- **Topic construction:** pick one event randomly, build:

```rust
Topic {
    seed: format!("In {year}, {short_text}"),
    context: Some(combined_page_extracts),
}
```

`prompt::build` then composes a system message ("You are a poet…") and a user message that uses `seed` as the subject and may quote `context` as background.

**Note:** Wikipedia's REST endpoint is rate-limited (200/s shared). Fine for a CLI; if phase 2 starts firing one per frame from Bevy, caching will be needed (deferred — see § 9).

## 6. Configuration

### 6.1 Layering

**Non-secret settings** (provider, model, source, temperature, etc.) — highest wins:

1. **CLI flags** — `--provider`, `--model`, `--source`, `--topic`, `--temperature`, `--max-tokens`, `--config`
2. **Config file** — `--config <path>`, else `./poet.ron`, else platform user-config dir (via `directories` crate: `%APPDATA%\rust-poet\poet.ron` on Windows), else…
3. **Built-in defaults**

**API keys** are a separate dimension, never part of the layered settings: read **only** from environment variables (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `MISTRAL_API_KEY`), resolved **lazily** for just the provider actually selected. You don't need keys for providers you aren't using, and keys never appear in the config file.

### 6.2 Sample `poet.example.ron`

```ron
(
    default_provider: "openai",
    default_source: "wikipedia",
    providers: {
        "openai":    (model: "gpt-4o-mini",          base_url: None),
        "anthropic": (model: "claude-haiku-4-5",     base_url: None),
        "mistral":   (model: "mistral-small-latest", base_url: None),
    },
    poem: (max_tokens: 400, temperature: 0.9),
    request_timeout_secs: 30,
)
```

`base_url: None` means "use the provider's standard endpoint." Setting it is how a user points the OpenAI-compatible client at Ollama or LM Studio later — no code change required.

### 6.3 CLI

```
rust-poet [OPTIONS]

OPTIONS:
  --provider <NAME>      openai | anthropic | mistral
  --model <NAME>         override default model for chosen provider
  --source <NAME>        wikipedia | random | fixed
  --topic <STRING>       required when --source=fixed; otherwise ignored
  --temperature <F>      0.0–2.0
  --max-tokens <N>
  --config <PATH>        path to poet.ron
  --print-config         dump effective merged config and exit (debugging)
  -v, --verbose          enable tracing at debug level
```

## 7. Error handling

### 7.1 Top-level error

```rust
#[derive(Debug, thiserror::Error)]
pub enum PoetError {
    #[error(transparent)] Provider(#[from] ProviderError),
    #[error(transparent)] TopicSource(#[from] TopicError),
    #[error(transparent)] Config(#[from] ConfigError),
}
```

- `ProviderError` — defined in § 4.3.
- `TopicError` — `Network`, `Decode`, `NoEventsForDate`, `BadInput`.
- `ConfigError` — `MissingApiKey { env_var }`, `FileParse { path, source }`, `UnknownProvider`, `UnknownSource`, `InvalidValue`.

### 7.2 Reporting

- `main.rs` uses `anyhow::Result` for top-level reporting.
- `MissingApiKey` is recognized explicitly so the user sees `error: OPENAI_API_KEY not set` rather than a wall of error chains.
- Exit code 1 on any error.

### 7.3 Deliberate non-features (with justification)

- **No retries.** `RateLimited { retry_after }` carries the data for a future retry policy without changing call sites or the trait.
- **No streaming.** A new method (or sibling trait) can return `impl Stream<Item = …>` later; existing `generate()` stays.
- **No caching.** Caching wraps `TopicSource` / `LlmProvider` impls — no trait change needed when added.

## 9. Testing strategy

### 9.1 Three layers

**Unit tests** (co-located `#[cfg(test)] mod tests`)

| Module | What's tested |
|---|---|
| `config` | layer merging, RON parse, CLI overrides win, missing-key surfaces correctly |
| `prompt` | given a `Topic`, output `Vec<Message>` shape matches expectation |
| `provider/openai_like` | request body serializes to expected JSON; success + error response fixtures parse correctly |
| `provider/anthropic` | same, with system-field/content-array quirks |
| `topic/wikipedia` | response fixture parses; empty-events case returns `TopicError::NoEventsForDate` |

These never touch the network. Fixtures are JSON string literals.

**Integration tests** (`tests/`)

- `MockLlmProvider` ships in `src/` behind a `test-utils` Cargo feature, so `tests/` and the future Bevy crate can both use it. (`FixedTopic` is the production `--topic` source — always available, not feature-gated.)
- **Pipeline glue test:** assemble a `Poet` from `MockLlmProvider` + `FixedTopic`, assert `generate()` returns the expected `Poem`.
- **HTTP round-trip test per provider** using `wiremock`: stand up a fake server, point the real client at `mock.uri()` via the `base_url` config knob, assert request headers/body and verify the response is parsed correctly. Same harness exercises 401 / 429 / 5xx error mapping.

**Live tests** — feature-gated `#[cfg(feature = "live-tests")]`, off by default.

One file per provider. Each test issues a single cheap call (`gpt-4o-mini` / `claude-haiku-4-5` / `mistral-small-latest`, `max_tokens: 50`). Skipped automatically when the corresponding env key isn't set. Never runs in CI; developer runs locally before merging changes that touch a provider's wire format.

### 9.2 Verification gates ("done" for the PoC)

- `cargo build  --manifest-path cross-platform/rust-poet/Cargo.toml` — clean
- `cargo test   --manifest-path cross-platform/rust-poet/Cargo.toml` — green
- `cargo clippy --manifest-path cross-platform/rust-poet/Cargo.toml -- -D warnings` — clean
- Manual smoke: `cargo run -- --topic rain --provider openai` produces a poem
- Manual smoke: `cargo run -- --source wikipedia --provider anthropic` produces a poem about today's date

## 10. Deferred work and how the design accommodates it

| Item | Lands in | How the design accommodates it |
|---|---|---|
| Bevy rendering with shader effects | Phase 2 | Library-first design; Bevy crate just `use rust_poet::*` |
| TTS / voice generation | Phase 2 or 3 | Likely a new trait `VoiceProvider`, parallel to `LlmProvider` |
| Streaming token output | Phase 2 | New trait method or sibling trait; `generate()` stays |
| Retry / backoff on 429 | Future | `RateLimited { retry_after }` already carries the data |
| Response caching (esp. Wikipedia) | Future | Caching wraps `TopicSource` / `LlmProvider` impls — no trait change |
| Style / mood / form modifiers | Future | New field on `PoemSettings`; threads through `prompt::build` only |
| News / RSS / themes-as-source | Future | New `TopicSource` impl per source |
| Additional providers (Gemini, Cohere, …) | Future | New `LlmProvider` impl per provider |
| Local LLM via Ollama / LM Studio | **Day one** | Point `base_url` at `http://localhost:11434/v1` in `poet.ron` — no code change |
